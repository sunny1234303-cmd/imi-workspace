# 서브에이전트 (Subagents) 완벽 가이드

> Claude에게 전문가 팀을 만들어주는 방법

## 서브에이전트란?

**서브에이전트**는 특정 업무만 전문으로 하는 Claude의 분신입니다.

### 왜 필요한가요?

| 상황 | 메인 Claude만 쓸 때 | 서브에이전트 쓸 때 |
|------|-------------------|------------------|
| 코드 리뷰 | 매번 "보안, 성능, 스타일 검토해줘" 설명 | `/code-reviewer` 한번으로 전문 리뷰 |
| 디버깅 | 일반적인 조언 | 체계적 디버깅 프로세스 자동 진행 |
| 컨텍스트 | 모든 내용이 한 대화에 쌓임 | 각 작업이 독립적으로 분리 |

## 기본 제공 서브에이전트

Claude Code에는 이미 3개의 서브에이전트가 내장되어 있습니다:

| 에이전트 | 모델 | 역할 |
|---------|------|------|
| **Explore** | Haiku (빠름) | 코드베이스 탐색, 파일 검색 (읽기 전용) |
| **Plan** | 상속 | 계획 모드에서 정보 수집 |
| **General-purpose** | 상속 | 복잡한 멀티스텝 작업 |

## 서브에이전트 만들기

### 방법 1: 대화형으로 만들기 (추천)

```
/agents
```

1. **Create new agent** 선택
2. **User-level** (개인용) 또는 **Project-level** (프로젝트용) 선택
3. **Generate with Claude** 선택 후 원하는 에이전트 설명
4. 도구, 모델, 색상 설정
5. 저장 완료!

### 방법 2: 파일로 직접 만들기

파일 위치:
- 개인용: `~/.claude/agents/에이전트이름.md`
- 프로젝트용: `.claude/agents/에이전트이름.md`

## 파일 구조 이해하기

```markdown
---
name: code-reviewer
description: 코드 품질과 보안을 검토하는 전문가
tools: Read, Glob, Grep
model: sonnet
---

여기에 에이전트가 할 일을 자세히 적습니다.
에이전트는 이 지시를 따라 작업합니다.
```

### 각 부분 설명

```yaml
---                           # ← 설정 시작 (이 선 필수!)
name: code-reviewer           # ← 에이전트 이름 (영어, 소문자, 하이픈만)
description: 코드 리뷰 전문가  # ← Claude가 언제 이 에이전트를 쓸지 판단하는 설명
tools: Read, Glob, Grep       # ← 사용할 수 있는 도구들
model: sonnet                 # ← 사용할 AI 모델
---                           # ← 설정 끝 (이 선 필수!)

(여기부터 실제 지시사항)
```

## 설정 옵션 상세

### 필수 옵션

| 옵션 | 설명 | 예시 |
|------|------|------|
| `name` | 에이전트 고유 이름 | `code-reviewer` |
| `description` | 언제 사용할지 설명 | "코드 품질 검토 전문가" |

### 선택 옵션

| 옵션 | 설명 | 예시 |
|------|------|------|
| `tools` | 허용할 도구 | `Read, Grep, Bash` |
| `disallowedTools` | 금지할 도구 | `Write, Edit` |
| `model` | AI 모델 | `sonnet`, `opus`, `haiku` |
| `permissionMode` | 권한 모드 | `default`, `acceptEdits` |
| `skills` | 함께 로드할 스킬 | `["code-review"]` |
| `hooks` | 자동 실행 스크립트 | (아래 예제 참조) |

### 사용 가능한 도구들

| 카테고리 | 도구 | 설명 |
|---------|------|------|
| **읽기** | `Read` | 파일 읽기 |
| | `Grep` | 텍스트 검색 |
| | `Glob` | 파일 패턴 검색 |
| **쓰기** | `Write` | 새 파일 작성 |
| | `Edit` | 파일 수정 |
| **실행** | `Bash` | 명령어 실행 |

### 권한 모드 (permissionMode)

| 모드 | 설명 | 언제 사용? |
|------|------|----------|
| `default` | 매번 권한 물어봄 | 안전하게 사용할 때 |
| `acceptEdits` | 파일 수정 자동 승인 | 신뢰할 수 있는 작업 |
| `dontAsk` | 권한 요청 자동 거부 | 읽기 전용 작업 |
| `bypassPermissions` | 모든 권한 자동 승인 | 완전 자동화 |
| `plan` | 계획 모드 (읽기만) | 탐색/분석 작업 |

### 모델 선택

| 모델 | 특징 | 추천 용도 |
|------|------|----------|
| `haiku` | 빠르고 저렴 | 간단한 작업, 탐색 |
| `sonnet` | 균형 잡힌 성능 | 대부분의 작업 |
| `opus` | 가장 강력 | 복잡한 추론, 창의적 작업 |
| `inherit` | 부모 대화와 동일 | 기본값 |

## 실전 예제

### 예제 1: 코드 리뷰어

```markdown
---
name: code-reviewer
description: 코드 품질, 보안, 유지보수성을 전문으로 검토하는 시니어 개발자
tools: Read, Grep, Glob, Bash
model: sonnet
---

당신은 시니어 코드 리뷰어입니다.

## 호출되면 할 일

1. `git diff`로 최근 변경사항 확인
2. 수정된 파일에 집중
3. 즉시 리뷰 시작

## 검토 체크리스트

- [ ] 코드가 읽기 쉬운가?
- [ ] 함수/변수 이름이 명확한가?
- [ ] 중복 코드가 없는가?
- [ ] 에러 처리가 적절한가?
- [ ] 비밀 정보(API 키 등)가 노출되지 않았는가?
- [ ] 입력값 검증이 되어 있는가?
- [ ] 테스트가 충분한가?
- [ ] 성능 문제는 없는가?

## 피드백 형식

우선순위별로 정리해서 알려주세요:

### 🔴 Critical (반드시 수정)
- 보안 취약점, 심각한 버그

### 🟡 Warning (수정 권장)
- 잠재적 문제, 코드 냄새

### 🟢 Suggestion (개선 제안)
- 더 나은 방법 제안

각 이슈에 대해 구체적인 수정 방법도 함께 알려주세요.
```

### 예제 2: 디버거

```markdown
---
name: debugger
description: 오류, 테스트 실패, 예상치 못한 동작을 해결하는 디버깅 전문가
tools: Read, Edit, Bash, Grep, Glob
model: sonnet
---

당신은 숙련된 디버거입니다.

## 문제 해결 프로세스

1. **증상 파악**
   - 에러 메시지 캡처
   - 스택 트레이스 분석

2. **재현 확인**
   - 문제 재현 단계 확인
   - 환경 정보 수집

3. **원인 격리**
   - 문제 발생 지점 특정
   - 로그 분석

4. **해결책 구현**
   - 최소한의 수정으로 해결
   - 부작용 확인

5. **검증**
   - 수정 후 테스트
   - 관련 기능 확인

## 보고 형식

각 이슈에 대해:
- 🔍 **근본 원인**: 왜 발생했는지
- 📋 **증거**: 어떤 근거로 판단했는지
- 🔧 **해결책**: 구체적인 코드 수정
- ✅ **테스트**: 어떻게 검증할지
- 🛡️ **예방**: 재발 방지 방법
```

### 예제 3: 한국어 번역가

```markdown
---
name: korean-translator
description: 영어 기술 문서를 자연스러운 한국어로 번역하는 전문가
tools: Read, Write, Edit
model: sonnet
---

당신은 기술 문서 번역 전문가입니다.

## 번역 원칙

1. **의역 우선**: 직역보다 자연스러운 한국어 표현
2. **용어 일관성**: 같은 용어는 같은 번역 유지
3. **문화적 적합성**: 한국 독자에게 친숙한 표현

## 용어 가이드

| 영어 | 한국어 |
|------|--------|
| repository | 저장소 |
| branch | 브랜치 |
| commit | 커밋 |
| pull request | 풀 리퀘스트 (PR) |
| merge | 병합 |
| deploy | 배포 |

## 작업 방식

1. 원문 파일 읽기
2. 섹션별로 번역
3. 전문 용어 일관성 확인
4. 최종 교정
```

### 예제 4: 읽기 전용 DB 쿼리 실행자 (Hooks 활용)

```markdown
---
name: db-reader
description: 읽기 전용 데이터베이스 쿼리를 실행하는 분석가
tools: Bash
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/validate-readonly-query.sh"
---

당신은 데이터베이스 분석가입니다.
SELECT 쿼리만 사용하여 데이터에 대한 질문에 답변합니다.

## 규칙
- SELECT 쿼리만 사용
- INSERT, UPDATE, DELETE 등 쓰기 작업 금지
- 민감 정보 조회 시 주의
```

검증 스크립트 (`./scripts/validate-readonly-query.sh`):
```bash
#!/bin/bash
INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // empty')

# 쓰기 작업 차단
if echo "$COMMAND" | grep -iE '\b(INSERT|UPDATE|DELETE|DROP|CREATE|ALTER|TRUNCATE)\b' > /dev/null; then
  echo "차단됨: 쓰기 작업은 허용되지 않습니다. SELECT만 사용하세요." >&2
  exit 2
fi

exit 0
```

## 서브에이전트 사용하기

### 자동 위임
Claude가 작업 내용과 에이전트 설명을 보고 자동으로 적절한 에이전트에게 위임합니다.

### 명시적 요청
```
code-reviewer 서브에이전트로 이 코드 리뷰해줘
debugger 에이전트한테 이 에러 분석 맡겨줘
```

### 포그라운드 vs 백그라운드

| 모드 | 특징 | 용도 |
|------|------|------|
| **포그라운드** | 대화 차단, 권한 물어봄 | 상호작용 필요한 작업 |
| **백그라운드** | 동시 실행, 권한 자동 거부 | 독립적 작업 |

## 저장 위치와 우선순위

| 위치 | 범위 | 우선순위 |
|------|------|---------|
| `--agents` CLI 옵션 | 현재 세션만 | 1 (최고) |
| `.claude/agents/` | 현재 프로젝트 | 2 |
| `~/.claude/agents/` | 모든 프로젝트 | 3 |
| 플러그인 agents/ | 플러그인 적용 범위 | 4 (최저) |

## 실무 팁

1. **설명(description)이 핵심**: Claude가 언제 이 에이전트를 쓸지 판단하는 기준
2. **도구 최소화**: 필요한 도구만 허용하여 안전성 확보
3. **모델 선택**: 간단한 작업은 `haiku`, 복잡한 작업은 `sonnet` 또는 `opus`
4. **테스트 먼저**: 새 에이전트는 간단한 작업으로 테스트 후 사용
